# レポート構成ガイドライン

レポートのセクション構成に関する詳細ガイドライン。

---

## 社内向けレポート構成

### 必須セクション

| セクション | 目的 | 目安分量 |
|---|---|---|
| エグゼクティブサマリー | 結論を3行以内で述べる | 3行以内 |
| 背景と目的 | なぜこの調査を行うかを説明する | 5〜10行 |
| 調査方法 | 調査範囲・手段・情報源を明示する | 5〜15行 |
| 調査結果 | 観点ごとに調査結果を整理する | 制限なし |
| 比較・分析 | 比較表を用いて分析する（比較調査の場合） | 表+解説 |
| 結論と推奨 | 結論と推奨アクションを明記する | 5〜10行 |
| Next Steps | 具体的なタスクをチェックリスト形式で記載する | 3〜10項目 |
| 参考文献 | 全ての情報源をリスト化する | 制限なし |

### オプションセクション

| セクション | 用途 |
|---|---|
| 用語集 | 専門用語が多い場合に追加する |
| 付録 | 詳細データやコード全文を掲載する |
| リスクと懸念事項 | 技術選定時にリスクを整理する |
| コスト試算 | SaaS導入やインフラ変更の場合に追加する |

### セクション作成ガイドライン

#### エグゼクティブサマリー

- 結論を先に述べる（結論ファースト）
- 技術的な詳細は含めない
- 読者がこのセクションだけで意思決定できる内容にする

```markdown
## エグゼクティブサマリー
> FastAPIを推奨する。パフォーマンス・開発体験・エコシステムの3観点で
> Djangoを上回り、マイクロサービス構成との親和性が高い。
> ただしORM機能が必要な場合はDjango + DRFも有力な選択肢である。
```

#### 背景と目的

- 調査に至った経緯を簡潔に述べる
- 調査の目的（意思決定、知識共有、技術選定 等）を明記する
- スコープ（何を調査対象とし、何を対象外とするか）を定義する

#### 調査方法

- 調査範囲を明記する
- 使用した情報源を列挙する
- 情報の信頼度基準を示す

```markdown
## 調査方法
- 調査範囲: Python Webフレームワーク（FastAPI, Django）
- 情報源:
  - 公式ドキュメント（信頼度: 高）
  - GitHub リポジトリ統計（信頼度: 高）
  - 技術ブログ・ベンチマーク記事（信頼度: 中）
- 調査期間: 2026年2月
```

#### 調査結果

- 観点ごとにサブセクションを設ける
- 各観点で事実と考察を分離する
- コード例やベンチマーク結果を適宜含める

#### 比較・分析

- 比較表は必ず含める
- 評価軸を明確にする
- 定性評価と定量評価を組み合わせる

```markdown
## 比較・分析
| 項目 | FastAPI | Django |
|---|---|---|
| パフォーマンス | ◎ 非同期対応 | △ 同期ベース |
| 学習コスト | ○ 中程度 | △ やや高い |
| エコシステム | ○ 成長中 | ◎ 成熟 |
```

#### 結論と推奨

- 調査結果に基づく結論を述べる
- 具体的な推奨アクションを提示する
- 条件付き推奨の場合は条件を明示する

#### Next Steps

- 具体的で実行可能なタスクを列挙する
- 優先度順に並べる
- 担当者・期限が分かれば記載する

## 対外向けレポート構成

### 必須セクション

| セクション | 目的 | 目安分量 |
|---|---|---|
| 表紙情報 | タイトル・日付・作成者 | 3〜5行 |
| エグゼクティブサマリー | 経営層が読む要約 | 5〜10行 |
| 背景 | ビジネス課題を説明する | 5〜10行 |
| 調査概要 | 調査手法を簡潔にまとめる | 5行程度 |
| 主要な発見事項 | 重要な結果をハイライトする | 制限なし |
| 比較・分析 | 図表中心で視覚的に示す | 表+解説 |
| 推奨事項 | 推奨アクションを提案する | 5〜10行 |
| ロードマップ | 実行計画を時系列で示す | 表形式 |
| 付録 | 技術詳細を補足する | 制限なし |

### セクション作成ガイドライン

#### エグゼクティブサマリー（対外向け）

- ビジネスインパクトを中心に述べる
- 技術用語は使わない（または注釈を付ける）
- 数値・効果を具体的に示す

#### 主要な発見事項

- 箇条書きで3〜5点にまとめる
- 各発見に対してビジネスへの影響を付記する
- 図表で視覚的に補強する

#### 推奨事項

- 優先度を明示する（高/中/低）
- 期待される効果を数値で示す
- 実現に必要なリソースを概算する

#### ロードマップ

- フェーズに分けて記載する
- 各フェーズの成果物を明記する
- マイルストーンを設定する

```markdown
## ロードマップ
| フェーズ | 期間 | 内容 | 成果物 |
|---|---|---|---|
| Phase 1 | 1ヶ月 | PoC実施 | 検証結果レポート |
| Phase 2 | 2ヶ月 | パイロット導入 | 運用手順書 |
| Phase 3 | 3ヶ月 | 本番展開 | 完全移行 |
```

## 共通ルール

### 文体

- 社内向け: である調（簡潔さ重視）
- 対外向け: ですます調（丁寧さ重視）

### 表記

- 数値は具体的に記載する（「速い」ではなく「レスポンスタイム50ms」）
- 略語は初出時にフルスペルを記載する
- 日付は `YYYY-MM-DD` 形式で統一する

### 参考文献

- Web検索で取得した情報には必ずURLを付記する
- 取得日を記載する
- 信頼度を明示する（公式ドキュメント > 技術ブログ > フォーラム投稿）
