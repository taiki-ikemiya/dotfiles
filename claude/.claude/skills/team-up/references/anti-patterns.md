# チーム編成のアンチパターン

避けるべき5つの編成パターン。これらを検出した場合、Team Proposalで警告を出す。

## AP1: 全員実装者（レビューなし）

**症状**: チーム全員がengineer系で、code-reviewerがいない。

**問題点**:
- バグや設計問題が見逃される
- コード品質が担保されない
- 「作る人と見る人を分ける」原則（P3）に違反

**対策**:
- 最低1人はレビュー担当をアサインする
- 実装者同士のクロスレビューも有効だが、専任レビュー者がいるほうが効果的

## AP2: 曖昧なファイル境界（上書き合戦）

**症状**: 複数メンバーが同一ファイルを編集する可能性がある。ファイル所有権が明示されていない。

**問題点**:
- 後から編集した人が先の変更を上書きする
- マージ困難なコンフリクトが発生する
- 作業のやり直しが頻発する

**対策**:
- spawn時にファイル所有権を明示する
- ディレクトリ単位で担当を分ける
- 共有ファイルは編集順序を決める

## AP3: リード不在の大チーム

**症状**: 3人以上のチームでリードモードが設定されていない。

**問題点**:
- 作業の重複や抜け漏れが発生する
- 成果物の統合が困難
- メンバー間の調整が不十分

**対策**:
- 3人以上のチームには必ずリードモードを設定する
- リードの役割を明確にする（coordinator / architect / facilitator）

## AP4: 過剰な人数

**症状**: タスクの規模に対してメンバーが多すぎる（5人以上）。

**問題点**:
- 調整コストが指数的に増大する
- 各メンバーの担当範囲が狭すぎて非効率
- コミュニケーションオーバーヘッドが大きい

**対策**:
- 最小有効チーム原則（P1）を適用する
- 大きなタスクはフェーズに分け、フェーズごとに小チームを組む
- 本当に5人以上が必要か再検討する

## AP5: コンテキスト不足のspawn

**症状**: メンバーをspawnする際に、タスクの背景や制約が十分に伝えられていない。

**問題点**:
- メンバーが的外れな作業を進める
- 手戻りが発生する
- 他メンバーとの連携がうまくいかない

**対策**:
- spawn promptに以下を含める:
  - ペルソナ定義（Spawn Prompt Template）
  - タスクの全体像と個別の担当範囲
  - ファイル所有権
  - 他メンバーとの連携ポイント
  - 完了条件
- 不足情報はユーザーに確認してからspawnする
