# チーム編成原則

team-upがチームを編成する際に適用する7つの原則。

## P1: 最小有効チーム

- **基本人数は2-4人**。タスクを達成できる最小人数で編成する
- 5人以上は例外的な場合のみ。人数が増えるほど調整コストが増大する
- 1人で十分なタスクにチームを組まない。team-upを使う必要がないなら使わない
- 人数の目安:
  - 2人: 実装 + レビュー、調査 + まとめ
  - 3人: フルスタック実装 + レビュー、調査 + 分析 + レポート
  - 4人: 複数領域の実装 + レビュー + ドキュメント
  - 5人以上: 大規模リファクタリング、複合的なプロジェクトなど例外のみ

## P2: ファイル所有権の分離

- **各ファイルの編集責任者を1人に限定する**
- 複数メンバーが同一ファイルを編集すると上書き合戦が発生する
- ファイル所有権はspawn時に明示する
- 共有ファイルが避けられない場合:
  - 編集順序を決める（先にAが編集 → 完了後にBが編集）
  - インターフェース定義を先に固め、実装は分離する
- ディレクトリ単位で所有権を分けるのが最も安全

## P3: 作る人と見る人を分ける

- **実装者とレビュー者は別のメンバーにする**
- 自分で書いたコードの問題には気づきにくい
- code-reviewerを明示的にアサインする
- ドキュメントにも同様の原則を適用する（writer + reviewer）
- レビュー者には実装のコンテキストを十分に共有する

## P4: リードの明確化

- **3人以上のチームにはリードモードを必ず設定する**
- リードが不在だと作業が断片化し、成果物の統合が困難になる
- リードモードの選択基準:
  - 実装中心 → coordinator
  - 設計判断含む → architect
  - 調査・比較・意思決定 → facilitator
- リードはspawnせず、自分自身の指揮スタイルとして適用する

## P5: タスク類型に応じた基本構成

- タスクの類型に応じて基本構成パターンがある（`task-team-mapping.md` 参照）
- 基本構成をベースに、タスク固有の要件で調整する
- 過去の成功パターンを再利用する

## P6: 調査は実装の前に

- **不確実性が高い場合、まず調査フェーズを設ける**
- 調査結果をもとに実装チームを編成し直すのが効果的
- 調査と実装を同時並行にしない（調査結果が実装に影響するため）
- 調査フェーズの成果物:
  - 技術選定の根拠
  - 実装方針のドラフト
  - リスクと制約の整理

## P7: 定期チェックイン

- **リードは各メンバーの進捗を定期的に確認する**
- ブロッカーの早期発見と解消が目的
- チェックインのタイミング:
  - タスク開始直後（方向性の確認）
  - 中間地点（進捗と問題の確認）
  - 完了前（成果物の統合準備）
- メンバーからの報告を待つだけでなく、能動的に確認する
