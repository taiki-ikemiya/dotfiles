# smart-commit スキル

Git の変更を分析し、適切な粒度で分割してコミットメッセージ案を提示する。
ユーザーの承認を得てからコミットを実行する。

---

## 実行フロー

### ステップ1: 変更の収集

以下のコマンドで現在の変更状態を把握する。

```bash
git status
git diff          # unstaged の変更
git diff --cached # staged の変更
git log --oneline -10  # 既存コミットのスタイルを確認
```

- untracked ファイルも `git status` で確認し、対象に含めるか判断する
- 既存コミットログから、プロジェクトのコミットメッセージが日本語か英語かを判定する
- staged と unstaged の両方がある場合は、その旨をユーザーに伝える

### ステップ2: 変更の分析とグループ化

収集した差分を論理的な単位にグループ化する。

#### グループ化の判断基準

- **type が異なるものは分割する**: feat と fix が混在する場合は別コミットにする
- **同一機能のファイル群はまとめる**: 同じ機能に属する実装ファイル・設定ファイル・型定義は1コミットにする
- **テストは実装と対にできる**: テスト追加が実装変更と1対1で対応するなら同じコミットに含めてよい
- **ドキュメント更新は独立させる**: README や CHANGELOG の更新は別コミットにする
- **設定ファイル変更はまとめる**: lint 設定、CI 設定など関連する設定変更は1コミットにする
- **リファクタリングは分離する**: 機能変更を伴わないリファクタは独立したコミットにする

#### 粒度の目安

- 1コミットあたりのファイル数は1〜10ファイル程度を目安にする
- 差分が大きすぎる場合はさらに分割を検討する
- 差分が1ファイルの1行修正でも、意味のある変更なら独立したコミットにする

### ステップ3: コミットメッセージ案の生成

各グループに対して Conventional Commits 形式のメッセージを生成する。

#### メッセージ形式

```
<type>(<scope>): <subject>

<body>

<footer>
```

- **type**: 変更の種類を示す（後述の一覧を参照）
- **scope**: 変更対象のモジュールやディレクトリ名（省略可）
- **subject**: 50文字以内、現在形で記述する
- **body**: 「何を変更したか」「なぜ変更したか」を説明する（必要な場合のみ）
- **footer**: Breaking Change や Issue 参照を記載する（必要な場合のみ）

#### type 一覧

| type | 用途 |
|------|------|
| feat | 新機能の追加 |
| fix | バグ修正 |
| refactor | 機能変更を伴わないコード改善 |
| docs | ドキュメントの変更 |
| test | テストの追加・修正 |
| chore | ビルドプロセスやツール設定の変更 |
| style | コードの意味に影響しないフォーマット変更 |
| perf | パフォーマンス改善 |
| ci | CI 設定の変更 |

#### 言語の選択

- プロジェクトの既存コミットが日本語なら日本語でメッセージを書く
- プロジェクトの既存コミットが英語なら英語でメッセージを書く
- 混在している場合や判断できない場合は英語を使用する

### ステップ4: コミット案の提示

以下の形式でユーザーに提示する。

```markdown
## コミット案

### Commit 1
- [ ] `feat(auth): JWT認証を実装`
  - src/auth/validator.ts
  - src/auth/middleware.ts

### Commit 2
- [ ] `fix(api): ユーザーエンドポイントのnullレスポンスを修正`
  - src/api/users.ts

### Commit 3
- [ ] `test(auth): JWT認証のユニットテストを追加`
  - tests/auth/validator.test.ts
```

提示時の注意事項:

- 各コミットに含まれるファイルパスを全て列挙する
- コミットの実行順序は依存関係を考慮する（基盤 → 機能 → テスト → ドキュメント）
- 変更行数が多い場合は概要も付記する

### ステップ5: ユーザー確認

ユーザーからの応答を自然言語で受け付ける。

#### 想定される応答パターン

- **全承認**: 「全部OK」「これでいい」「実行して」→ 全コミットを順に実行する
- **部分承認**: 「1だけ」「1と3をお願い」→ 指定されたコミットのみ実行する
- **メッセージ修正**: 「2のメッセージを〜に変えて」→ 修正して再提示する
- **グループ変更**: 「1と3をまとめて」「2を分割して」→ 再グループ化して再提示する
- **全却下**: 「やめて」「キャンセル」→ 何も実行しない
- **追加情報要求**: 「差分を見せて」→ 該当ファイルの diff を表示する

修正依頼があった場合は、修正後のコミット案を再度提示してから実行する。

### ステップ6: コミットの実行

承認されたコミットを順番に実行する。

```bash
# Commit 1
git add src/auth/validator.ts src/auth/middleware.ts
git commit -m "feat(auth): JWT認証を実装"

# Commit 2
git add src/api/users.ts
git commit -m "fix(api): ユーザーエンドポイントのnullレスポンスを修正"
```

実行時の注意事項:

- 各コミットの実行結果を表示する
- エラーが発生した場合は即座に停止してユーザーに報告する
- 全コミット完了後に `git log --oneline` で結果を確認表示する

## 安全策

### 禁止事項

- `git commit --amend` はユーザーが明示的に要求しない限り使用しない
- `git push --force` はユーザーが明示的に要求しない限り使用しない
- `git reset --hard` は使用しない
- `git clean -f` は使用しない
- `.env` やクレデンシャルファイルなど機密情報を含むファイルはコミットしない

### 警告を出すべき状況

- staged と unstaged の変更が混在している場合
- コミット対象に `.env`、`credentials`、`secret`、`token` を含むファイル名がある場合
- 未追跡の大きなバイナリファイルがある場合
- コミットメッセージの subject が50文字を超える場合
- 直前のコミットと同じファイルを再度変更している場合（amend の方が適切な可能性）

### 実行前チェックリスト

1. 全ての対象ファイルが存在することを確認する
2. コミットメッセージが Conventional Commits 形式に準拠していることを確認する
3. 対象ファイルに機密情報が含まれていないことを確認する
4. ユーザーの承認を得ていることを確認する

## body の記述ガイドライン

body が必要なケースと不要なケースを判断する。

### body を付ける場合

- 変更理由が subject だけでは伝わらない場合
- 複数の関連ファイルをまとめて変更している場合
- Breaking Change を含む場合
- 既知のバグや Issue に対応する場合

### body のフォーマット

```
<type>(<scope>): <subject>

<何を変更したかの説明>

<なぜ変更したかの説明>

<footer（Breaking Change や Issue 参照）>
```

body は72文字で折り返す。箇条書きを使ってもよい。

## エッジケースの対処

### 変更がない場合

`git status` で変更がないことを確認したら、その旨を伝えて終了する。
空コミットは作成しない。

### 変更が1ファイル1行の場合

分割の必要がないため、1コミットで提示する。

### 巨大な変更（50ファイル以上）の場合

- まずディレクトリ単位で大まかなグループを提示する
- ユーザーと相談しながら詳細な分割を決定する
- 一度に全てをコミットしようとしない

### マージコンフリクトがある場合

コミット操作を行わず、コンフリクト解決を先に促す。

### ブランチが main/master の場合

直接 main/master にコミットしてよいか確認する。
